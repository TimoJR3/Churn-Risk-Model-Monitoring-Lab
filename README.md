# Churn Risk & Model Monitoring Lab

Reproducible churn modeling project for Data Science intern / Junior Data
Scientist roles. The repository demonstrates a realistic ML demo service:
synthetic churn data, feature engineering, model validation, saved artifacts,
FastAPI inference, PostgreSQL prediction logs, PSI drift checks, and a Russian
Streamlit dashboard.

It is intentionally a portfolio MVP, not a production deployment.

```text
synthetic data -> preprocessing -> training -> artifacts
-> FastAPI online/batch scoring -> PostgreSQL prediction logs
-> monitoring endpoints -> Streamlit dashboard
```

## Business Problem

A subscription product wants to identify users with elevated churn risk before
they leave. The model estimates churn probability from product activity,
payment friction, support interactions, plan type, and recency signals.

The output is meant to help prioritize retention analysis and outreach. It is
not a direct business decision engine.

## Dataset and Features

The dataset is synthetic and generated by
`app.ml.generate_synthetic_data`. It simulates a subscription product snapshot
with a binary target:

- Target: `churn`, where `1` means the user churned.
- User/profile fields: `user_id`, `signup_date`, `country`, `plan_type`,
  `monthly_fee`.
- Activity fields: `days_active_last_30`, `sessions_last_30`,
  `avg_session_duration`, `feature_usage_score`, `last_login_days_ago`.
- Friction fields: `support_tickets_last_30`, `payments_failed_last_90`.

Feature engineering in `app.ml.features` adds:

- `activity_score`;
- `payment_risk_score`;
- `engagement_level`;
- `days_since_signup`;
- `usage_per_session`;
- `support_intensity`.

Missing values and a small number of outliers are intentionally injected so the
preprocessing pipeline has realistic edge cases.

## Modeling Approach

The training script compares baseline scikit-learn models:

- Logistic Regression with balanced class weights;
- Random Forest with balanced class weights;
- HistGradientBoostingClassifier.

The preprocessing pipeline handles numeric and categorical features:

- numeric median imputation + standard scaling;
- categorical most-frequent imputation + one-hot encoding;
- `user_id` and `signup_date` are dropped before model fitting.

Training saves:

- `artifacts/trained_model.pkl`;
- `artifacts/preprocessor.pkl`;
- `artifacts/metrics.json`;
- `artifacts/feature_importance.csv`;
- `docs/model_card.md`.

## Validation Strategy

The project uses a stratified train/validation split with default
`test_size=0.2` and `random_state=42`.

Model selection uses `StratifiedKFold` cross-validation on the training split.
The main selection metric is ROC-AUC, with F1 as a secondary signal.

Validation metrics saved in `metrics.json` include:

- ROC-AUC;
- F1;
- precision;
- recall;
- confusion matrix;
- classification report.

Accuracy is not the primary metric because churn is commonly imbalanced:
predicting mostly "no churn" can look accurate while missing many at-risk users.

## Inference API

FastAPI exposes online scoring through:

- `GET /health`;
- `POST /predict`;
- `GET /model/metadata`.

`POST /predict` accepts one user payload, applies the saved preprocessor and
model, and returns:

- `churn_probability`;
- `churn_prediction`;
- `risk_band`: `low`, `medium`, or `high`;
- `threshold`;
- `model_version`;
- `model_artifact_name`;
- `explanation`.

If artifacts are missing, the API returns HTTP 503 with the training command to
run. It does not retrain models during requests.

## Batch Scoring

`POST /predict/batch` accepts a list of user payloads and returns:

- `row_count`;
- `items`, each with the same response fields as `/predict`.

The max batch size is configured through `MAX_BATCH_SIZE` and defaults to 100.
Demo payloads are stored in:

- `data/sample/predict_sample.json`;
- `data/sample/predict_batch_sample.json`.

## Prediction Logging

`POST /predict` writes prediction logs to PostgreSQL when
`SAVE_PREDICTIONS=true`.

The logging layer stores:

- generated `request_id`;
- `user_id_hash` instead of raw `user_id`;
- probability, prediction, risk band, threshold, model version;
- sanitized input features;
- timestamp.

Recent logs are available through:

- `GET /predictions/recent?limit=20`.

For local API-only demos without PostgreSQL, set `SAVE_PREDICTIONS=false`.

## Model Monitoring

Monitoring is intentionally small, transparent, and testable:

- `GET /monitoring/summary` summarizes recent prediction logs;
- `POST /monitoring/drift` calculates PSI for one numeric feature;
- `POST /monitoring/quality` calculates ROC-AUC, precision, recall, and F1
  from labels and scores.

The Streamlit dashboard is fully localized in Russian and consumes the API for
single prediction, batch scoring, monitoring summary, and model metadata.

## PSI Drift Explanation

Population Stability Index compares an expected numeric feature distribution
with a current or actual distribution.

In this project:

- `stable`: PSI `< 0.1`;
- `warning`: `0.1 <= PSI < 0.25`;
- `drift`: PSI `>= 0.25`.

PSI is useful as a simple drift signal, but it is univariate. It does not prove
that the model is wrong, and it does not detect all multivariate distribution
changes. Here it is a monitoring demo, not a full alerting system.

## Quickstart

### Windows PowerShell

```powershell
py -3.11 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install -r requirements.txt -r requirements-dev.txt
Copy-Item .env.example .env

python -m app.ml.generate_synthetic_data --n-users 2000 --seed 42
python -m app.ml.preprocessing --source csv --test-size 0.2
python -m app.ml.training --source csv --n-splits 3
```

Run API locally without PostgreSQL logging:

```powershell
$env:SAVE_PREDICTIONS="false"
uvicorn app.api.main:app --reload
```

Run the dashboard:

```powershell
streamlit run dashboard/app.py
```

### macOS / Linux

```bash
python3.11 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt -r requirements-dev.txt
cp .env.example .env

python -m app.ml.generate_synthetic_data --n-users 2000 --seed 42
python -m app.ml.preprocessing --source csv --test-size 0.2
python -m app.ml.training --source csv --n-splits 3
```

Run API locally without PostgreSQL logging:

```bash
export SAVE_PREDICTIONS=false
uvicorn app.api.main:app --reload
```

Run the dashboard:

```bash
streamlit run dashboard/app.py
```

### Docker Compose

Docker Compose starts API, dashboard, and PostgreSQL. In this mode prediction
logging is enabled by default.

```bash
cp .env.example .env
docker compose up --build
```

Services:

- API: <http://localhost:8000>
- Swagger UI: <http://localhost:8000/docs>
- Dashboard: <http://localhost:8501>
- PostgreSQL: `localhost:5432`

## What This Project Demonstrates for DS Roles

- Framing churn prediction as a binary classification problem.
- Building a reproducible synthetic dataset for portfolio work.
- Separating feature engineering, preprocessing, training, inference, and
  monitoring code.
- Comparing baseline models with stratified cross-validation.
- Reporting ROC-AUC, precision, recall, F1, and confusion matrix.
- Saving and reusing model artifacts for inference.
- Exposing online and batch scoring through typed API schemas.
- Logging predictions with a privacy boundary around `user_id`.
- Implementing a small PSI drift check with edge-case tests.
- Presenting results through a dashboard for non-technical review.

## Limitations

- Dataset is synthetic; metrics are not business benchmarks.
- No real production traffic or SLA is simulated.
- The default threshold `0.5` is not cost-optimized.
- There is no temporal validation split because the dataset is a generated
  snapshot.
- PSI monitoring is request-driven and simplified.
- There is no scheduled monitoring job, alert routing, model registry, or
  automated retraining.
- Dashboard is a local demo UI, not an authenticated operational console.
